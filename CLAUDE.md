## 1. 作るもの（概要）

医科・DPC のオンライン請求（毎月: 作成→事前点検→請求確定→審査→結果配信・入金）を支援する2機能。

1. **エラーコード検索・一覧**: 受付・事務点検ASP結果に出るエラーコード（例 `L4428`）を検索・絞り込みし、区分（L1〜L4/L48/L7）・対応区分（再請求要否）・対象点数表（医科/DPC）・原因・対応目安を表示。
2. **増減点連絡書 分析**: 審査結果（減点）の傾向を、診療科別・医師別・事由別（A〜F）・区分別・月次推移で可視化。自院CSVの読み込みに対応。

試作 `reception_support_tool.jsx` に両機能の動作版が入っている。**これを移植のたたき台にする**（データ・ロジック・UIをほぼ流用可）。

## 4. データモデル

### 4.1 エラーコードマスタ（`public/data/error-codes.json`）

1件のスキーマ:

```jsonc
{
  "code": "L4428",              // エラーコード
  "level": "L4",                // L1 | L2 | L3 | L4 | L48 | L7
  "targets": ["医科", "DPC"],   // 適用する点数表（医科のみ/DPCのみ/両方）
  "cat": "算定ルール",           // 形式・構造/資格・保険者/公費/DPCコーディング/算定ルール/傷病名/金額・負担金
  "must": false,                // 受付不能ASP（●）= 保険医療機関の修正が必須か
  "msg": "届出と異なる施設基準を算定しています…",   // エラー又は確認事項
  "cause": "届出と異なる施設基準の診療が記録された…" // エラー原因
}
```

**level の意味（支払基金の定義。UIの色分け・トリアージの根拠）**

| level | 意味 | 対応区分 |
|---|---|---|
| L1（1000番台） | 医療機関単位のエラー | 再請求が必要 |
| L2（2000番台） | レセプト単位のエラー | 再請求が必要 |
| L3（3000番台） | レセプト内エラー・要確認 | 返戻となる要確認 |
| L4（4000番台, L48除く） | レセプト内エラー・要確認 | 査定/返戻/正当となる要確認 |
| L48（4800番台） | 連絡のみ（正常受付） | 再請求は不要 |
| L7（7000番台）※医科 | 算定ルールのコンピュータチェック | 査定/返戻/正当となる要確認 |

**マスタの作り方（重要）**
- 出典は支払基金「受付・事務点検ASPに係るチェック一覧」の **医科版・DPC版**（PDF）。歯科・調剤・訪看も同様の体系。
  - 医科: `https://www.ssk.or.jp/seikyushiharai/iryokikan/download/index.files/checkichiran_ika.pdf`（約570件, L1〜L4/L48/L7）
  - DPC: `https://www.ssk.or.jp/seikyushiharai/iryokikan/download/index.files/checkichiran_dpc.pdf`（約930件, 診断群分類・包括評価・コーディングデータ等のDPC固有を含む）
- **注意**: 同じ内容のチェックでも**コード番号が医科とDPCで異なる場合がある**（例: 生年月日の暦チェックは医科 `L2133` / DPC `L2174`、保険者番号形式は医科 `L2167` / DPC `L2181`）。本番マスタは**各公式一覧の正式コードをそのまま格納**し、`targets` で点数表を区別する。試作では実装簡略化のため共通概念に集約している箇所がある（フッター注記参照）ので、本番で正規化すること。
- 更新は**診療報酬改定ごと**（および支払基金の版更新ごと）。PDF→JSON 変換のスクリプトを用意すると運用が楽（`scripts/build-master.ts` など。表形式PDFのパース）。`must`（●）と `level`（種別列）と `cat`（内容から分類）を必ず正しく反映する。

### 4.2 増減点連絡書レコード（分析の入力）

1件のスキーマ:

```jsonc
{
  "ym": "2025-08",     // 診療年月
  "dept": "内科",       // 診療科
  "doctor": "DR001",   // 医師（★氏名ではなくコード推奨。個人情報保護）
  "reason": "D",       // 増減点事由 A〜F
  "kubun": "検査",      // 診療区分（検査/投薬/注射/画像診断/処置/手術/医学管理/リハビリ/入院料/DPC包括 等）
  "item": "腫瘍マーカー", // 項目名（診療行為・医薬品等）
  "ten": 120           // 減点点数（正の整数。金額は ten×10 円）
}
```

**増減点事由（reason）**: A=適応 / B=過剰・重複 / C=その他医学的 / D=算定要件 / F=事務的（固定点数・計算誤り等）。

**取込CSVの想定列（UTF-8）**: `診療年月, 診療科, 医師, 事由, 区分, 項目名, 増減点数`
- 生の増減点連絡書CSVをそのまま入れない。上記の**非識別化・集計済み抽出**を入力とする。
- 取込処理は**指定列のみ読み、患者氏名・レセプト番号・被保険者記号番号・受給者番号などの個人識別列は読み込まない**（試作の `onFile` と同じ方針）。

## 5. 機能仕様

### 5.1 エラーコード検索・一覧

- 検索: コード（`L4428`）または内容キーワード（`msg`/`cause` 部分一致）。
- 絞り込み: 区分（L1〜L4/L48/L7）チップ、分類セレクト、対象（医科/DPC/共通）セレクト、修正必須（●）チェック。
- 一覧: コード＋対象／区分バッジ／必須（●）／エラー内容。行クリックで詳細。
- 詳細: コード・区分バッジ・分類・対象・修正必須・**対応区分**（levelから: 再請求が必要／要確認／再請求は不要）・エラー内容・原因・対応の目安。
- （拡張候補）ASP結果リスト（テキスト/CSV）を貼り付け→出現コードを自動で本マスタに突合し、優先度順（●必須→再請求→要確認→連絡のみ）に一覧化。対応状況（未着手/対応中/完了）をローカルで管理。

### 5.2 増減点連絡書 分析

- KPI: 減点点数・減点金額（×10円）・査定件数・1件平均・最新月の前月比。
- グラフ（Recharts）: 月次推移（線）、診療科別（横棒）、事由別 A〜F（円）、区分別（縦棒）。
- テーブル: 減点点数 上位項目、医師別。
- フィルタ: 診療年月・診療科。トレンドは全期間・診療科フィルタ反映。
- データ: 初期はサンプル生成（試作の `makeSample`）。自院CSVを読み込むと差し替え。
- （拡張候補）項目/グラフのクリックで該当明細に絞り込むドリルダウン、事由×区分のクロス集計、査定率（対請求）の算出、月次レポート出力。

### 5.3 増減点データの累積（方式A：利用者が管理する累積ファイル）

**目的**: 月次トレンドを見るには複数月の累積が必要。非識別化した増減点データを月ごとに追記して貯める。累積ファイルは**アプリ／リポジトリの外**（院内の管理された共有フォルダ等）に置き、**利用者が管理**する。GitHub Pages は静的（サーバ・DBなし）のため、蓄積はサーバ側ではなくこの「利用者管理ファイル」で行う。

**累積ファイルの形式**: 往復（読込→追記→書出）の確実性のため **JSON** を正とする（例 `assessment-cumulative.json`）。Excelで見たい場合の CSV 書き出しは付加機能として別途用意してよい。

```jsonc
{
  "schemaVersion": 1,
  "updatedAt": "2026-01-20T10:00:00+09:00",
  "periods": ["2025-08", "2025-09", "2025-10"],   // 取り込み済みの診療年月
  "records": [
    { "ym": "2025-08", "dept": "内科", "doctor": "DR001", "reason": "D", "kubun": "検査", "item": "腫瘍マーカー", "ten": 120 }
    // ... 4.2 のレコードスキーマに準拠（医師はコード）
  ]
}
```

**月次の運用フロー**
1. 「累積ファイルを開く」で前月までの `assessment-cumulative.json` を共有フォルダから選択して読み込む。
2. 「当月分を追加」で当月の非識別化CSV（`診療年月/診療科/医師/事由/区分/項目名/減点点数`）を取り込む。
3. アプリが**診療年月の重複を判定**：当月の診療年月がすでに累積にあれば、二重計上を防ぐため「置き換える／取り込まない」を確認（既定=置き換え）。
4. マージ後、「累積を書き出す」でダウンロード。**同名で共有フォルダに保存し、旧ファイルを上書き**。
5. 画面に累積の対象期間（最古〜最新）と件数を表示。

**保存の実装（静的サイトの制約）**
- 基本: ブラウザのダウンロード（Blob）で書き出し→利用者が手動で共有フォルダへ保存。読み込みはファイル選択。ブラウザはパス指定でフォルダへ直接書けないため、「開く／保存」を明示操作にする。
- 拡張（対応ブラウザのみ）: **File System Access API**（`showOpenFilePicker` / `showSaveFilePicker` と `FileSystemFileHandle`）を使うと、同じファイルを開いて上書き保存でき再ダウンロード不要。Chromium系・HTTPS必須。未対応時はダウンロード方式にフォールバック。

**重複・整合性**
- キーは診療年月（バッチ単位）。同一診療年月の再取り込みは既定で置き換え。
- `schemaVersion` を持たせ将来の形式変更に備える。`updatedAt` / `periods` はマージ時に再計算。

**個人情報の扱い（6章と整合）**
- 累積ファイルは**非識別化データ**（患者識別情報を含まない、医師はコード化）。
- 累積ファイルは**リポジトリにコミットしない**。`.gitignore` に `assessment-cumulative*.json` を追加（7章の `.gitignore` 参照）。保存先は院内のアクセス権を設定した共有フォルダ。
- さらに機微さを下げるなら、明細ではなく**月次集計値のみ**（診療科×事由×区分の件数・点数合計）を累積する派生方式も可（この場合、明細ドリルダウンは不可になる）。

**UI 追加**
- ツールバーに「累積ファイルを開く」「当月分を追加」「累積を書き出す」ボタンと、対象期間・件数の表示。既存のサンプル／自院データ切替と共存し、累積読込時は「累積データ」表示に。

**実装メモ**
- マージ（置き換え）: `const merged = [...cumulative.filter(r => r.ym !== newYm), ...newRecords]`。
- 書き出し: `new Blob([JSON.stringify(data, null, 0)], { type: "application/json" })` → `URL.createObjectURL` でダウンロード。
- File System Access API: `const w = await handle.createWritable(); await w.write(blob); await w.close();`。

## 6. 個人情報・セキュリティ（重要）

### 6.1 現行設計での保護の考え方

**結論: この設計なら個人情報は保護できる。理由は「アプリが公開でも、扱うデータは公開されない」から。**

- GitHub Pages が配信するのは**アプリのコード（HTML/JS/CSS）だけ**。アプリは利用者のブラウザ内で動く。
- 利用者がファイル選択で読み込んだCSVは**ブラウザ内（PapaParseとReactのstate）で処理**され、**どのサーバにも送信されない／保存されない**。ページを閉じれば消える。
- したがって、**リポジトリが公開（public）であっても、アップロードした患者データが公開されることはない**。

**危険になるのは次のパターン。これらを設計で避ける。**

1. 実データのCSV/Excelを**リポジトリにコミット**してしまう → 公開され、履歴にも残る。→ `.gitignore` で `*.csv` 等を除外。データは `public/` に置かない。
2. データを**外部サービス/API/解析ツールに送信**する → 送信先に残る。→ 本アプリは外部送信を一切実装しない（分析はクライアントサイドのみ）。埋め込み解析（GA等）も入れない。
3. 共有PCで **localStorage/sessionStorage 等にPIIを永続化** → 端末に残る。→ PIIは永続化しない（メモリ保持のみ）。
4. **医師名**は個人情報 → 可能な限り**医師コード化**。
5. 生の増減点連絡書CSV（患者氏名・レセプト番号・被保険者記号番号等を含む）をそのまま扱う → 入力は**非識別化・集計済み**に限定し、取込は指定列のみ読む。

**運用・規程面**: 医療機関では「医療情報システムの安全管理に関するガイドライン」等（いわゆる3省2ガイドライン）に留意。**患者データを外部の公開Webサイト上で取り扱うことは院内規程で認められない場合がある**。技術的にブラウザ内処理でも、院内のルール確認を前提とすること。

### 6.2 GitHub Pages（公開リポジトリ）の注意点

- 無料の user/organization Pages および public リポジトリの Pages は**URLを知れば誰でもアクセスできる公開サイト**（アクセス制御なし）。ただし前述の通り、**公開されるのはコードのみ**。
- private リポジトリからの Pages 公開・アクセス制御は **GitHub Team / Enterprise** の機能（無料枠では不可）。

### 6.3 GitHub Pages（公開）が不都合な場合の代案

| # | 代案 | 個人情報の扱い | 向き |
|---|---|---|---|
| 1 | **非識別化・集計済みデータのみ扱う（推奨）** | 患者識別情報を入力しない。医師はコード化。 | エラー参照＋増減点分析の両方を public Pages で安全に運用 |
| 2 | **機能分割** | エラー参照（PIIなし）＝public Pages、増減点分析（データ扱う）＝院内へ | 公開してよい機能だけ外に出す |
| 3 | **完全ローカル運用** | データも端末外に出ない | ビルド済み `index.html` を院内PCで開く／配布。ホスティング自体なし |
| 4 | **院内イントラの静的ホスティング** | 院内網内で完結 | 社内Webサーバ・ファイルサーバに `dist` を置く |
| 5 | **プライベートPages** | GitHub認証でアクセス制御 | GitHub Team/Enterprise 契約がある場合 |

**推奨方針**: エラーコード参照は PII を扱わないので public Pages で問題なし。増減点分析は「非識別化・集計済みデータのみ（代案1）」を原則とし、患者単位・生データを扱う可能性があるなら「院内ローカル/イントラ（代案3・4）」に置く。判断は院内の情報セキュリティ規程に従う。

## 7. デプロイ（GitHub Pages）

**`vite.config.ts`**（project pages の場合、`base` にリポジトリ名を設定）:

```ts
import { defineConfig } from "vite";
import react from "@vitejs/plugin-react";

export default defineConfig({
  plugins: [react()],
  base: "/reception-support/", // 例: https://<user>.github.io/reception-support/
});
```

**`.github/workflows/deploy.yml`**:

```yaml
name: Deploy to GitHub Pages
on:
  push:
    branches: [main]
permissions:
  contents: read
  pages: write
  id-token: write
concurrency:
  group: pages
  cancel-in-progress: true
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: dist
  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
```

- リポジトリの Settings → Pages → Source を **GitHub Actions** に設定。
- SPA を Pages で使う場合、404対策として `dist/404.html` に `index.html` をコピーする（Vite の場合はビルド後にコピーするステップを足すか、ハッシュルーティングを使う）。本アプリはタブ切替のみでルーティング不要なので基本不要。

## 8. マスタ更新の運用（診療報酬改定）

- 改定・版更新のたびに、支払基金の医科版・DPC版チェック一覧PDFから `public/data/error-codes.json` を再生成。
- `level`・`must`（●）・`targets`・`cat` の付与を必ず見直す。差分は Pull Request でレビュー。
- 分析側の `区分`・`事由` の定義も改定で変わり得るため確認。

## 9. 既存試作の活用と実装ステップ

1. `reception_support_tool.jsx` を `src/` に取り込み、`ErrorSearch` / `AssessmentAnalysis` を機能ごとにファイル分割。
2. 内蔵のエラーコード配列を `public/data/error-codes.json` に切り出し、`fetch` で読み込むよう変更。`level`/`reason` 定義は `src/lib/` へ。
3. Tailwind をビルドに正式導入（試作のクラスはほぼ流用可）。任意値クラスも使用可に。
4. CSV取込（`src/lib/csv.ts`）を、指定列のみ読む方針で堅牢化。医師コード運用に合わせる。
5. GitHub Actions を設定し、`base` を合わせて Pages 公開（PIIを扱わない範囲で）。増減点分析でデータを扱うなら 6.3 の方針を適用。
6. マスタを医科・DPCの全件へ拡張（PDF→JSON 変換）。
